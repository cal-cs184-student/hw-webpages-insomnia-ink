Task 1: Drawing Single-Color Triangles

Rasterization

In this task, I implemented the rasterize_triangle function in rasterizer.cpp to render single color triangles. To rasterize a triangle, we need to determine which pixels should be filled based on their position relative to the triangle’s edges. To do so, we can first determine the bounding box that contains the entire triangle. Then iterate over all pixels in this bounding box and use a point-in-triangle test to decide whether to fill the pixel with the given color.

  1. Bounding Box Calculation - Find the min and max x/y coordinates of the triangle's vertices.
  1. Point-in-Triangle Test (Half-Plane Method) - A triangle can be represented as the intersection of three half-planes, each defined by one of its edges. For a given point (x,y), we check if it is inside all three half-planes by computing the edge functions (signed area tests).
  1. Filling the Pixel - If the point lies inside the triangle (i.e., passes all three half-plane tests), we call fill_pixel(x, y, color) to color the pixel.

Efficiency

Instead of checking every pixel in the frame, we limit the scan to only the bounding box of the triangle. The half-plane test (edge function method) evaluates three conditions per pixel, which is a constant-time operation O(1) per pixel. This means that even if we were to check every sample within the bounding box, our approach wouldn’t add unnecessary computations beyond those needed.

Both approaches (this implementation and a simple bounding box checker) will examine every sample in the bounding box. So, because this algorithm only processes pixels within the bounding box, it is inherently no worse than an approach that blindly checks every sample within the bounding box.

Since this algorithm only iterates over the bounding box, and for each pixel it performs at most three simple evaluations (one per edge), its efficiency is bounded by the bounding box size. This ensures that it is never slower than a bounding box scan, as it does not evaluate anything outside of it, and that it is at least as efficient as an approach that blindly checks each sample within the bounding box. Thus, our implementation achieves the same complexity as a full bounding box scan, making it at least as good as a naïve approach that checks every sample in the bounding box without optimization.

To optimize the performance beyond sampling every pixel in the bounding box, I implemented the following improvements:

I used a Tiled Triangle Traversal approach which processes blocks of pixels instead of individual samples.

  1. Start with computing the Bounding Box like before
  1. Tiled Triangle Traversal (Block-Based Rasterization)- Now, instead of checking every pixel, I divided the bounding box into small tiles (e.g., 4x4 pixel blocks). For each tile, I tested whether all pixels are inside the triangle, all are outside, or the block is partially covered.
    • If fully inside, I filled the entire block (early inclusion). If fully outside, I skipped the block (early rejection). Otherwise, I performed per-pixel tests within that tile.
  1. Edge Function Test - for each pixel inside partially covered tiles, I used the edge function to determine whether the point was inside the triangle. The edge function is based on the cross-product of two triangle edges and helps efficiently classify points.
  1. Filling Pixels - If the pixel is inside the triangle (based on edge function tests), I called fill_pixel(x, y, color) to rasterize it.

Better efficiency reasons:

1. Avoid Redundant Edge Function Computations -Instead of recalculating the edge function for every pixel, I incrementally updated the values as we moved across the grid.

2. Block-Wise Memory Access - Processing pixels in 4x4 or 8x8 tiles allowed for better cache efficiency, reducing memory access time.

3. Early-In and Early-Out - This method skips large areas of pixels using “early in” and “early out” optimizations. It minimizes per-pixel operations by testing blocks first before individual samples. This prevented unnecessary point-in-triangle tests when an entire block was inside or outside the triangle.

Task 2: Antialiasing by Supersampling

Aliasing mainly happens because we are not taking enough samples per pixel, leading to jagged edges. In supersampling, instead of sampling just one point per pixel, we sample multiple points. By taking multiple samples per pixel and averaging them, we effectively smooth out high-frequency details. The final pixel color is a blend of the sampled colors making edges look much more natural.

Algorithm

  1. Sample Buffer Expansion - Instead of storing just one color per pixel, we now store sample_rate colors per pixel in sample_buffer. The buffer is resized to (width * s) * (height * s), where s = sqrt(sample_rate).This ensures that every output pixel has an s × s grid of subpixels.
  1. Subpixel Sampling (Rasterizing Triangles with Supersampling) - For each pixel in the bounding box of the triangle, we divide it into an s × s grid of subpixels. The edge function test determines if a subpixel is inside the triangle. If it is, the corresponding subpixel in sample_buffer is colored.
  1. Resolve to Framebuffer (Downsampling) - At the end of rendering, each output pixel is computed by averaging all its s × s subpixels.This smooths jagged edges because pixels near a triangle’s boundary have mixed colors.

Data Structures

  • sample_buffer stores all supersamples instead of just pixels.
  • rasterize_triangle() processes all subpixels instead of the pixel center.
  • resolve_to_framebuffer() averages the s×s samples per pixel.

Use of Supersampling

Without supersampling, each pixel is either fully inside or fully outside the triangle. This causes jagged edges (aliasing) because a pixel can only be on or off.

With supersampling, multiple samples per pixel capture partial triangle coverage. When downsampled, the output pixel blends the colors based on coverage. This creates smooth edges instead of jagged, stair-step patterns.

Modifications to the Rasterization Pipeline

To support supersampling, the following modifications were made:

  1. Expanded sample_buffer Storage - Previously, it stored width * height colors (one per pixel). Now it stores width * height * s * s colors (one per subpixel, where s is the sqrt(sample_rate)).
  1. Modified rasterize_triangle()- Instead of testing one sample per pixel, we now test s × s subpixels.Each subpixel's exact position is computed for precise coverage.
  1. Updated resolve_to_framebuffer()- Instead of copying colors directly, we now average s × s samples per pixel.This ensures a smooth downsampled image.
  1. Adjusted fill_pixel() for Points & Lines so points and lines are drawn into all subpixels within a pixel.

Task 4: Barycentric coordinates

Barycentric coordinates let us express any point inside a triangle as a weighted combination of the vertices, with weights summing to 1. These weights, α, β, and γ, represent the influence of each vertex on the point, and they also let us interpolate colors (or any other attributes) smoothly across the triangle.

Example of a svg file with a single triangle with one red, one green, and one blue vertex:

When rendering a triangle where each vertex has a different color, we can use barycentric coordinates to compute the color at any point inside the triangle. If each vertex has a color (R, G, B), we can compute the interpolated color at any point inside the triangle using: Color(P)= α⋅Color(A)+ β⋅Color(B)+ γ⋅Color(C)

Since these weights vary smoothly across the triangle, the color transitions gradually rather than abruptly. In a triangle with red, green, and blue vertices, this interpolation blends the colors inside naturally, ensuring a seamless gradient. Each pixel’s final color is a weighted sum of the three vertex colors, preserving a smooth and continuous transition across the surface.

Task 5: "Pixel sampling" for texture mapping

Pixel sampling is the process of determining the color of a pixel in a rendered image by referencing a texture image. Instead of assigning a single color to a triangle, texture mapping applies an image (texture) to a 3D or 2D surface. This is done by mapping the screen-space coordinates to texture-space (UV) coordinates and then sampling the texture at those UV locations.

To apply texture mapping, I started by computing the barycentric coordinates for each pixel inside a triangle, relative to the three triangle vertices. Then using these barycentric coordinates, I interpolated the corresponding (u, v) texture coordinates. Using the computed (u, v) coordinates, I sampled the texture at the appropriate location using either nearest neighbor or bilinear interpolation, depending on the chosen sampling method. The fetched texture color was then used to determine the final pixel color.

This implementation ensured that the texture was mapped correctly to the triangle, allowing for smooth color transitions when using bilinear filtering.

Nearest Neighbor vs. Bilinear Sampling

Nearest Neighbor Sampling - This method finds the closest texel (texture pixel) to the computed (u, v) coordinates and uses its color directly. It is fast but often results in blocky, pixelated textures, especially when the texture is scaled up.

Bilinear Sampling - Instead of picking the nearest texel, bilinear interpolation blends the four nearest texels using a weighted average based on the fractional parts of the UV coordinates. This produces smoother textures, reducing pixelation and aliasing artifacts.

Bilinear sampling is preferred when smooth transitions are needed, while nearest neighbor is faster but less visually appealing.

And also if the texture contains gradients or fine details, such as text, soft shadows, or smooth color transitions or if the texture is applied to angled or curved surfaces, where aliasing is more apparent. In these cases, nearest sampling produces jagged and unnatural results, while bilinear sampling helps preserve smoothness and realism.

Task 6: "Level Sampling" with mipmaps for texture mapping

Level sampling is a technique used in texture mapping to reduce aliasing by selecting the appropriate texture resolution based on the pixel's screen-space coverage. Instead of always sampling from the highest-resolution texture, we use precomputed mipmaps—a series of downscaled versions of the original texture—to pick the most suitable level of detail. This prevents excessive detail from being compressed into small screen-space areas, which can cause artifacts like aliasing and shimmering in distant objects. By dynamically choosing the mipmap level, we can balance texture quality and performance.

Implementation

To implement level sampling for texture mapping, I first computed the barycentric coordinates of each pixel inside the triangle to interpolate its UV texture coordinates. Since level sampling requires understanding how texture coordinates change across the screen, I also computed the UV coordinates for neighboring pixels and by measuring the difference between these UV coordinates and the original pixel’s UV, I estimated how much the texture is stretched or compressed in screen space.

Using these differences, I determined the appropriate mipmap level by estimating how much detail is needed for the given pixel.

  • If the level sampling mode was set to L_ZERO, the program always sampled from the highest-resolution mipmap level.
  • If L_NEAREST was selected, I rounded the computed mipmap level to the nearest integer and sampled from that level.
  • For L_LINEAR, I performed trilinear interpolation, blending between two adjacent mipmap levels to create a smoother transition between texture resolutions.

After selecting the mipmap level, I used either nearest-neighbor or bilinear interpolation based on the pixel sampling mode to fetch the final color from the texture. By implementing this, I ensured that textures appear crisp at close distances while avoiding aliasing at far distances.

Tradeoffs Between Different Sampling Techniques

We now have control over three sampling techniques: pixel sampling, level sampling, and the number of samples per pixel. Each method has its own tradeoffs in terms of speed, memory usage, and anti-aliasing effectiveness:

  • Pixel Sampling (P_NEAREST, P_LINEAR)
    • Fastest because it only looks at the texture at a fixed level.
    • Uses minimal memory but can cause aliasing when textures are scaled down.
    • Bilinear filtering helps smooth pixel transitions but doesn't prevent aliasing.
  • Level Sampling (L_ZERO, L_NEAREST, L_LINEAR)
    • Uses precomputed mipmaps, which slightly increase memory usage but greatly improve performance.
    • Reduces aliasing in distant textures by using coarser versions of the texture.
    • L_NEAREST picks a single mipmap level, while L_LINEAR blends between two levels for smoother transitions.
  • Supersampling (Higher Sample Rate)
    • Most computationally expensive, since it samples multiple sub-pixels per pixel.
    • Requires a large amount of memory for storing and processing the additional samples.
    • Most effective at eliminating aliasing and producing smooth results, but not practical for real-time applications like games.
