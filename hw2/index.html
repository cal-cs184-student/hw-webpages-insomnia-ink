Section 1: Bezier Curves and Surfaces

Part 1: Bezier Curves with 1D de Casteljau Subdivision

De Casteljau‚Äôs algorithm is a recursive method for evaluating Bezier curves by performing linear interpolation between control points. Given n+1 control points and a parameter t (ranging from 0 to 1), the algorithm computes intermediate points at each level using: Pi = (1 - t) * Pi + t * Pi+1; until only one final point remains, which lies on the curve.

Steps:

  1. Start with control points
  1. Using linear interpolation create new points between each point pair
    newPoint = (1 - t) * p1 + t * p2;
  1. Repeat on the new points until only one point remains. Each step reduces the number of points by 1, so eventually one single point remains.

Part 2: Bezier Surfaces with Separable 1D de Casteljau

De Casteljau‚Äôs algorithm, originally used for evaluating 1D Bezier curves, can be extended to Bezier surfaces by applying it twice - once for each parameter, u and v.

  1. First pass (along rows, using u):
    • Each row of control points can be treated as a 1D Bezier curve.
    • Use de Casteljau‚Äôs algorithm to compute intermediate points at u along each row. (compute the u-interpolated result for each row using evaluate1D(...).)
    • This reduces the n√ón grid into a 1D list of n points.
  1. Second pass (along columns, using v):
    • The 1D list of points from step 1 forms another Bezier curve in the vertical (v) direction.
    • Apply de Casteljau‚Äôs algorithm again at v to get the final interpolated point. (run evaluate1D(...) again to compute the final interpolated point at v.)

Implementation overview:

  • evaluateStep(...) does one step of de Casteljau subdivision on a 1D list of 3D points
  • evaluate1D(...) does full evaluation of a 1D Bezier curve. Recursively applies evaluateStep(...) until only one point remains, which lies on the Bezier curve.
  • evaluate(...) combines two 1D Bezier evaluations to compute a Bezier surface.

Section 2: Triangle Meshes and Half-Edge Data Structure

Part 3: ¬†Area-Weighted Vertex Normals

By default, meshes are rendered with flat shading, meaning each triangle has a single normal, leading to sharp lighting transitions. To make a surface look smoother, we compute vertex normals by averaging the normals of the surrounding faces. To make the shading look more natural, instead of just averaging normals, we weight each normal by the area of its corresponding face. This helps large triangles contribute more to the vertex normal than small ones.

To compute smooth vertex normals for Phong shading, I iterated through all the faces connected to a vertex and computed their area-weighted normals.

For each adjacent face, I:

  1. Extracted the three vertices using the half-edge data structure.
  1. Computed two edge vectors from the triangle.
  1. Calculated the face normal using the cross product of the two edges.
  1. Weighted the normal by the triangle‚Äôs area, which is proportional to |cross_product|.
  1. Summed all area-weighted normals and normalized the result to obtain a unit normal.

This approach ensures that larger triangles contribute more to the final normal than smaller ones, resulting in a more accurate and smooth shading effect. Finally, I normalized the accumulated normal to maintain a consistent magnitude.

Part 4: Edge Flip

The edge flip operation modifies the local connectivity of a mesh by replacing the shared edge between two adjacent triangles with a new diagonal. Given an edge e0e_0e0 shared by two triangles, we reassign the connectivity of the affected half-edges, vertices, edges, and faces while ensuring the overall mesh remains manifold and valid.

Implementation Steps:

  1. Retrieve Half-Edges and Elements - Extract half-edges (h0-h5) associated with the edge to be flipped and the two adjacent faces. Also identify the outer half-edges (h6-h9) to ensure continuity. Extract the four vertices (a, b, c, d) involved in the flip, and retrieve the two faces that will be modified and put them in variables.
  1. Boundary Check - If either adjacent face is a boundary, the function returns early without modifying the mesh.
  1. Reassign Half-Edges - Update next, twin, face, edge, and vertex pointers to reconfigure the two triangles around the new diagonal and maintain proper connectivity.
  1. Similarly update Faces, Edges, and Vertex pointers appropriately to reflect the new form.
  1. Return the Updated Edge Iterator

Part 6: Loop Subdivision for Mesh Upsampling

I implemented Loop subdivision following the suggested approach:

  1. Compute new positions for original vertices - Used the Loop subdivision formula to update the position of each old vertex based on the weighted sum of its neighboring vertices. Stored the updated positions in newPosition to prevent overwriting values during iteration. Used Vertex::isNew = false to keep track of original vertices.
  1. Compute new positions for edge midpoints - For boundary edges, simple midpoint averaging. For interior edges, the Loop subdivision formula used adjacent triangle vertices. Stored the computed values in Edge::newPosition.
  1. Split all original edges - Stored all original edges in a vector before modifying the mesh. Iterated through this list and split each edge, creating a new vertex at the midpoint. Assigned newPosition of the new vertex based on precomputed edge values. Tracked which edges were newly created using Edge::isNew.
  1. Flip new edges that connect an old and new vertex - Identified edges that connect an old vertex and a new vertex and stored them in a list. Iterated through this list and flipped each edge to maintain a well-connected topology.Ensured each edge was flipped only once by setting Edge::isNew = false after flipping.
  1. Update all vertex positions - After splitting and flipping, finalized vertex positions using the precomputed newPosition values.

Debugging Process : For debugging, I mainly used ./meshedit to step through subdivision iterations and inspect intermediate results. During testing, the mesh sometimes crashed after subdivision. Tracked the issue to uninitialized pointers when updating half-edges after splitting. The fix was to carefully update half-edge connections for each new vertex.

Observations on Mesh Behavior

1. Effect on Sharp Corners and Edges- After repeated subdivisions, sharp features (corners and edges) become rounded. This happens because Loop subdivision smooths the mesh, distributing vertices more evenly. To preserve sharp features, I pre-split edges near sharp corners before running subdivision. This increased edge density, preventing excessive smoothing.

2. Asymmetry in the Cube - Running Loop subdivision on dae/cube.dae led to slight asymmetry after several iterations. This happens because the original cube lacks uniform vertex connectivity, causing small imbalances to amplify over multiple subdivisions.

Pre-Processing for Symmetry - To make the subdivision more symmetric, I pre-processed the cube by pre-splitting some edges to create an "X" pattern in each square face. This balanced the vertex positions, reducing asymmetry and preserving the cube‚Äôs shape better after multiple subdivisions.
